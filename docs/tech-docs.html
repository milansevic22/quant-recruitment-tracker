How The App Works

A plain-English technical walkthrough of the dashboard, Firebase data flow, protected API routes, and careers-page scanner.

One-Sentence Summary

The app is a Next.js dashboard that reads recruitment data from Firestore when Firebase is configured, falls back to sample data when it is not, and uses protected server routes to seed data or scan public careers pages.

Request Flow

  1. A user opens the dashboard at /.
  2. The React dashboard checks whether browser-safe Firebase config exists.
  3. If config exists, it reads companies, jobs, and scanRuns from Firestore.
  4. If config is missing or the read fails, it displays realistic sample data instead of crashing.
  5. The dashboard renders summary cards, the discovered jobs table, tracked companies, and the latest scan result.

Review-Mode Interactions

The deployed dashboard is useful before Firebase credentials are added. A reviewer can search jobs, filter by company, filter by role type, filter by status, open external job links, refresh the data source, run a review scan, and change a job status locally for the current browser session.

Persisted status updates are supported through the protected PATCH /api/jobs/:jobId/status route. The public dashboard keeps the dropdown local so the admin secret is never exposed in browser code.

Automation Center

The dashboard includes an automation panel that shows watched sources, indexed postings, intended scan cadence, scanner health, and the fetch-match-record pipeline. In review mode, the Run review scan button adds newly detected sample roles locally so the reviewer can see the monitoring loop without needing live Firebase credentials.

In production mode, the same concept maps to the protected /api/scan route, Firestore writes, and optional Vercel Cron scheduling.

Live Scheduled Scanning

The repository includes vercel.json with a daily cron schedule for /api/cron/scan. When deployed on Vercel, that route can run the same scanner automatically and write new jobs into Firestore.

  1. Vercel Cron sends a scheduled GET request to /api/cron/scan.
  2. The route verifies the Authorization header against CRON_SECRET.
  3. The route initializes Firebase Admin using server-only credentials.
  4. The scanner checks active public careers pages, extracts matching roles, deduplicates jobs, and writes scan results to Firestore.
  5. The dashboard reads Firestore and updates with the latest jobs and scan-run history.

Verified ATS Sources

The scanner now includes first-class Greenhouse adapters for Jane Street and Optiver US. Those endpoints return public JSON job data, which is more reliable than scraping JavaScript-rendered careers pages. The scanner still falls back to safe HTML parsing for firms where no stable public ATS feed has been added yet.

Email Alerts

Newly inserted jobs can trigger a Resend email alert. This is enabled by setting RESEND_API_KEY, ALERT_EMAIL_TO, and optionally ALERT_EMAIL_FROM in Vercel. If those variables are missing, scans still run and persist jobs, but no email is sent.

Frontend

  • app/page.tsx renders the dashboard.
  • components/dashboard.tsx owns loading, fallback, error, and display states.
  • components/summary-card.tsx renders the metric cards.
  • components/status-badge.tsx renders consistent status labels.
  • Tailwind CSS provides the responsive internal-tool layout.

Firebase

  • lib/firebase/client.ts initializes the browser-safe Firebase client.
  • lib/firebase/admin.ts initializes the server-only Firebase Admin SDK.
  • Client reads use public NEXT_PUBLIC_FIREBASE_* values.
  • Server writes use private Firebase service-account variables.
  • No real secrets are committed to the repository.

Admin Routes

Route Purpose Protection
POST /api/seed Writes initial companies and sample jobs into Firestore. Requires x-admin-secret matching ADMIN_API_SECRET.
POST /api/scan Runs the public careers-page scanner and stores scan results. Requires x-admin-secret matching ADMIN_API_SECRET.
PATCH /api/jobs/:jobId/status Persists a review status of new, seen, applied, or ignored. Requires x-admin-secret matching ADMIN_API_SECRET.

Scanner Behavior

  1. The scanner loads active tracked companies from Firestore.
  2. If Firestore has no company records yet, it uses the initial company list.
  3. For verified ATS sources, it reads public JSON from Greenhouse.
  4. For other companies, it makes one gentle public request to the careers URL.
  5. It parses links from the HTML and checks for role keywords such as quant, trading, intern, graduate, research, and software.
  6. It creates job records with deterministic IDs so duplicate title and URL combinations are avoided.
  7. It stores a scanRuns record with companies checked, jobs found, new jobs added, and errors.
  8. If Resend is configured, it emails newly inserted jobs.

Fallback Mode

The sample-data mode is intentional. It lets a reviewer open the app immediately without Firebase credentials, while still seeing the dashboard experience, filters, external links, and local review controls. Admin-only persistence remains available through protected server routes. The sample index now includes a broader set of quant firms and hedge funds so the product feels closer to a real monitoring workflow.

Deployment Mode

In production, Vercel hosts the Next.js app. Firebase stores the recruitment data. Admin routes run on the server and can be called manually or by Vercel Cron.

Safety Boundaries

The scanner does not bypass logins, bot protection, CAPTCHAs, paywalls, or website restrictions. If a careers page cannot be read cleanly, the scanner records an error and moves on.

What To Say In The Presentation

This page is the best place to explain the technical side. Say that the frontend is reviewable immediately with sample data, Firebase is the intended source of truth, and protected server routes handle the sensitive write operations.
